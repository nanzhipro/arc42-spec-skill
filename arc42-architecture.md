# arc42 架构沟通表达框架：深度调研报告与通用规约

> 调研日期：2026-04-27  
> 适用范围：任意软件系统、平台、语言、技术栈、组织形态下的架构文档、技术方案、重构方案、演进方案、关键决策说明  
> 主要来源：arc42 Template Overview、arc42 Documentation、arc42 Download、arc42 Q42 Quality Model

---

## 0. 摘要

arc42 是一套面向软件架构沟通与文档化的通用模板。它不绑定某种编程语言、运行平台、架构风格、图形记法、组织流程或工具链。它回答的是一个更基础的问题：**一份架构文档应该稳定表达哪些信息，才能让相关方理解、评审、实现、演进和接手一个系统。**

arc42 把架构文档拆成 12 类信息：

1. 引言与目标
2. 架构约束
3. 上下文与范围
4. 解决方案策略
5. 构建块视图
6. 运行时视图
7. 部署视图
8. 横切概念
9. 架构决策
10. 质量要求
11. 风险与技术债
12. 术语表

这 12 类信息构成一套技术无关的表达框架。无论系统是后端服务、移动应用、桌面客户端、嵌入式系统、数据平台、AI 应用、前端应用、基础设施、插件系统或企业内部工具，都可以用同一套结构说明：目标是什么、边界在哪里、由什么构成、如何运行、如何部署、如何保证质量、为什么这么决策，以及还有哪些风险。

采用 arc42 的目的不是把文档写长，而是让架构表达变得：

- **结构化**：信息有稳定位置，不靠作者个人习惯。
- **可沟通**：不同角色能快速找到自己关心的内容。
- **可评审**：目标、约束、结构、流程、质量、风险都有明确检查点。
- **可追溯**：关键决策和取舍被记录下来。
- **可演进**：文档能随系统变化持续维护。
- **可交接**：新成员能理解系统为什么这样组织，而不仅知道代码在哪里。

## 2. arc42 的核心思想

### 2.1 架构文档首先服务沟通

arc42 的出发点是沟通，而不是形式。架构文档应帮助相关方理解：

- 系统为什么存在。
- 系统解决什么问题。
- 系统不解决什么问题。
- 哪些约束限制了架构选择。
- 系统边界在哪里。
- 系统和外部世界如何交互。
- 系统内部如何分解。
- 关键运行时流程如何发生。
- 软件如何部署到运行环境。
- 哪些规则、模式和概念贯穿多个部分。
- 哪些重要决策已经做出，为什么做。
- 质量目标如何验证。
- 已知风险和技术债在哪里。
- 核心术语如何统一。

因此，arc42 的“模板”本质是一套架构沟通协议：让作者和读者都知道某类信息应该出现在哪里。

### 2.2 相关方驱动

架构文档不是只写给架构师或开发者。不同相关方关注的信息不同：

| 相关方 | 典型关注点 |
| --- | --- |
| 业务负责人 / 产品负责人 | 目标、范围、风险、交付影响、关键取舍 |
| 开发者 | 模块职责、接口、依赖、运行时流程、代码映射 |
| 测试 / QA | 质量场景、边界条件、异常路径、验收标准 |
| 运维 / SRE | 部署结构、运行单元、监控、告警、回滚、容量 |
| 安全 / 合规 | 信任边界、数据流、权限、审计、风险 |
| 数据 / 算法团队 | 数据来源、数据生命周期、质量要求、接口契约 |
| 管理者 | 进度风险、组织约束、决策后果、演进路线 |
| 新成员 | 上下文、术语、结构、关键决策、代码入口 |

通用规约：任何架构文档都应先回答：**谁会读这份文档，他们要据此做什么决策或行动。**

### 2.3 质量目标驱动架构

架构不是功能列表的展开。很多架构决策来自质量目标：性能、可靠性、安全、可维护性、可演进性、可用性、可运维性、成本、可测试性、可移植性等。

| 质量目标 | 典型架构影响 |
| --- | --- |
| 性能 / 低延迟 | 缓存、异步、预热、分片、批处理、就近访问 |
| 可靠性 | 超时、重试、熔断、隔离、降级、恢复、幂等 |
| 安全 | 身份认证、权限控制、加密、审计、最小权限、数据脱敏 |
| 可维护性 | 模块边界、接口契约、命名、测试、依赖管理 |
| 可演进性 | 版本化、兼容策略、插件化、抽象边界、迁移路径 |
| 可运维性 | 指标、日志、追踪、健康检查、配置、回滚 |
| 成本 | 资源利用率、弹性、容量规划、构建和运维成本 |
| 可测试性 | 测试替身、可观测状态、确定性、可控依赖 |

通用规约：架构文档必须列出最高优先级质量目标，并把它们转化为可验证场景。

### 2.4 多视图分离

复杂系统无法用单一视图讲清楚。arc42 将架构表达拆成多个互补视图：

| 视图 | 回答的问题 |
| --- | --- |
| 上下文视图 | 系统边界在哪里，外部交互对象是谁 |
| 构建块视图 | 系统静态结构如何分解 |
| 运行时视图 | 关键场景中构建块如何协作 |
| 部署视图 | 软件构件如何映射到运行环境 |
| 横切概念 | 哪些规则和机制贯穿多个构建块 |

这些视图彼此独立又互相映射。上下文视图定义边界，构建块视图说明结构，运行时视图说明行为，部署视图说明运行形态，横切概念说明跨模块规则。

### 2.5 决策可追溯

系统结构不是自然生成的，它来自一系列取舍。arc42 第 9 章要求记录重要架构决策，避免未来只看到“结果”，看不到“原因”。

通用规约：凡是影响系统结构、质量属性、外部接口、关键依赖、数据模型、部署模型、安全边界、团队协作方式的决策，都应被记录。

### 2.6 风险显式化

成熟的架构文档不只描述理想路径，也描述风险。风险不是方案的瑕疵，而是工程现实的一部分。第 11 章把风险和技术债放到明面上，便于评审、排序、验证和管理。

通用规约：高影响风险必须具备触发条件、影响、缓解措施、验证方式和 owner。

### 2.7 模板可裁剪

arc42 是通用模板，不是僵硬流程。轻量方案可以裁剪，复杂方案应完整。裁剪的原则是：**减少不必要细节，但不隐藏关键风险和关键决策。**

通用规约：裁剪章节时必须说明“不适用原因”，避免读者无法判断是遗漏还是刻意省略。

## 3. arc42 的 12 章结构

### 3.1 总览

| 章节 | 名称 | 核心问题 | 典型交付物 |
| --- | --- | --- | --- |
| 1 | 引言与目标 | 为什么做，目标是什么，谁关心 | 背景、目标、非目标、质量目标、相关方 |
| 2 | 架构约束 | 哪些限制不可忽略 | 技术、组织、合规、平台、流程、资源约束 |
| 3 | 上下文与范围 | 系统边界在哪里 | 上下文图、外部系统表、输入输出、接口表 |
| 4 | 解决方案策略 | 顶层方案如何达成目标 | 核心策略、关键技术取舍、质量目标映射 |
| 5 | 构建块视图 | 静态结构如何分解 | Level 1/2/3 构建块图、职责、接口、依赖 |
| 6 | 运行时视图 | 关键场景如何发生 | 主流程、异常流程、状态机、时序图 |
| 7 | 部署视图 | 软件如何运行和分布 | 运行单元、部署单元、环境、网络、存储 |
| 8 | 横切概念 | 跨模块规则是什么 | 安全、错误、日志、配置、并发、数据一致性等 |
| 9 | 架构决策 | 为什么这样选 | ADR、决策表、后果说明 |
| 10 | 质量要求 | 如何证明架构满足质量 | 质量树、质量场景、指标、验收标准 |
| 11 | 风险与技术债 | 哪些问题需要管理 | 风险表、技术债、缓解计划 |
| 12 | 术语表 | 关键词如何统一 | 术语、缩写、领域词、同义词 |

### 3.2 第 1 章：引言与目标

**目的**：说明系统或方案的背景、目标、非目标、最高优先级质量目标和相关方。

**必须表达**：

- 背景与问题陈述。
- 业务目标或工程目标。
- 明确非目标。
- 3 到 5 个最高优先级质量目标。
- 相关方及其关注点。

**通用写法**：

```markdown
## 1. 引言与目标

### 1.1 背景与问题
说明系统或方案为什么需要存在。

### 1.2 目标
列出本方案要达成的目标。

### 1.3 非目标
列出本方案明确不解决的问题。

### 1.4 质量目标
| 优先级 | 质量目标 | 可验证场景 |

### 1.5 相关方
| 相关方 | 关注点 | 需要从文档获得什么 |
```

**常见错误**：

- 只写功能需求，不写质量目标。
- 目标太泛，例如“提升系统能力”，无法验证。
- 没有非目标，导致评审范围不断扩大。
- 没有相关方，导致文档不知道为谁服务。

### 3.3 第 2 章：架构约束

**目的**：列出限制架构选择自由度的因素。

**约束分类**：

| 类型 | 示例 |
| --- | --- |
| 技术约束 | 既有系统、语言、框架、协议、数据模型、兼容性 |
| 运行环境约束 | 云环境、终端环境、浏览器、操作系统、硬件、网络 |
| 组织约束 | 团队技能、交付周期、协作边界、代码归属 |
| 合规约束 | 安全、隐私、审计、数据主权、行业法规 |
| 流程约束 | 发布流程、审批流程、变更窗口、CI/CD |
| 资源约束 | 成本、容量、人力、时间、依赖团队可用性 |
| 约定约束 | 命名、目录、日志、错误码、版本策略、文档规范 |

**通用规约**：

- 每条约束必须写明影响。
- 约束不是背景材料，而是会影响架构选择的条件。
- 与目标冲突的约束必须显式说明。

### 3.4 第 3 章：上下文与范围

**目的**：明确系统边界、外部交互对象、输入输出和接口。

arc42 通常区分业务上下文和技术上下文：

| 类型 | 内容 |
| --- | --- |
| 业务上下文 | 业务参与者、业务事件、业务输入输出、领域接口 |
| 技术上下文 | 协议、通道、进程、网络、存储、消息、文件、API |

**典型交付物**：

- 系统上下文图。
- 外部参与者表。
- 外部系统表。
- 外部接口表。
- 输入输出映射。
- 范围内 / 范围外清单。

**通用写法**：

```markdown
## 3. 上下文与范围

### 3.1 系统边界
用图或文本说明系统黑盒。

### 3.2 外部参与者与系统
| 外部对象 | 类型 | 输入 | 输出 | 接口 / 通道 |

### 3.3 范围内

### 3.4 范围外
```

**常见错误**：

- 还没有定义系统边界，就开始描述内部模块。
- 外部系统只写名字，不写输入、输出和接口。
- 忽略外部接口的质量要求，例如安全、超时、幂等、吞吐、兼容性。

### 3.5 第 4 章：解决方案策略

**目的**：用简洁方式说明塑造架构的顶层策略。

**典型内容**：

- 系统分解方式。
- 核心技术路线。
- 数据策略。
- 集成策略。
- 达成质量目标的关键手段。
- 面对关键约束的应对方式。

**通用规约**：

- 第 4 章写顶层策略，不展开实现细节。
- 策略应能映射第 1 章质量目标和第 2 章约束。
- 细节放到构建块视图、运行时视图、部署视图、横切概念和架构决策中。

**推荐表格**：

| 质量目标 / 约束 | 解决方案策略 | 细节位置 |
| --- | --- | --- |
| 性能 | 异步处理、缓存、批处理、资源隔离 | 第 5/6/8 章 |
| 可靠性 | 超时、重试、熔断、降级、恢复 | 第 6/8/10 章 |
| 安全 | 身份校验、权限边界、审计、数据保护 | 第 8/9/10 章 |

### 3.6 第 5 章：构建块视图

**目的**：描述系统静态结构，也就是系统由哪些构建块组成，以及它们之间的职责和依赖。

构建块可以是任何技术层级的结构单元：

- 子系统
- 服务
- 应用
- 模块
- 包
- 组件
- 类
- 函数库
- 运行单元
- 数据处理单元
- 基础设施组件

arc42 采用黑盒 / 白盒层级展开：

| 层级 | 含义 |
| --- | --- |
| Level 1 | 把整体系统作为白盒，展示内部一级黑盒构建块 |
| Level 2 | 展开 Level 1 中关键构建块的内部结构 |
| Level 3 | 展开 Level 2 中复杂、风险高或需要评审的构建块 |

**黑盒描述模板**：

| 字段 | 说明 |
| --- | --- |
| 名称 | 构建块名称 |
| 职责 | 该构建块负责什么 |
| 对外接口 | 提供给其他构建块或外部系统的能力 |
| 依赖 | 它依赖哪些构建块、数据或外部系统 |
| 质量要求 | 性能、安全、可靠性、可维护性等特殊要求 |
| 映射位置 | 代码、配置、部署单元、仓库、包或服务名 |
| 风险 | 复杂点、技术债、演进难点 |

**通用规约**：

- 每份架构文档至少应包含 Level 1 构建块视图。
- 只展开重要、复杂、风险高、变化频繁的构建块。
- 构建块命名应稳定，并与代码、部署或领域语言建立映射。
- 构建块视图描述结构，不负责描述完整时序。

### 3.7 第 6 章：运行时视图

**目的**：描述关键场景中，构建块实例如何交互。

适合记录：

- 主成功路径。
- 关键异常路径。
- 启动、停止、升级、恢复。
- 状态流转。
- 安全敏感流程。
- 性能敏感流程。
- 数据一致性流程。
- 跨系统协作流程。

**通用场景选择规则**：

必须优先写：

- 相关方最关心的业务主流程。
- 最容易出事故的异常流程。
- 最能体现架构取舍的流程。
- 影响质量目标的流程。

可以省略：

- 没有架构意义的普通增删改查。
- 简单局部函数调用。
- 与关键质量目标无关的 UI 操作细节。

### 3.8 第 7 章：部署视图

**目的**：说明软件构建块如何映射到实际运行环境。

部署视图中的“部署”是通用概念，不限于服务器。它可以表示：

- 机器、虚拟机、容器、集群。
- 浏览器、移动端、桌面端、嵌入式设备。
- 进程、线程、运行时、函数、任务。
- 包、插件、扩展、库、二进制产物。
- 数据库、缓存、消息队列、对象存储。
- 网络区域、边缘节点、客户环境、离线环境。

**通用规约**：

- 只要系统跨进程、跨机器、跨运行单元、跨网络边界或涉及特殊运行环境，就必须写部署视图。
- 部署视图必须说明构建块到运行产物的映射。
- 部署视图应说明环境差异、配置差异、容量和运行依赖。

### 3.9 第 8 章：横切概念

**目的**：集中描述跨多个构建块的规则、模式、策略和技术概念。

常见横切概念：

| 概念 | 示例内容 |
| --- | --- |
| 安全 | 认证、授权、信任边界、数据保护、审计 |
| 错误处理 | 错误码、异常传播、重试、降级、补偿 |
| 日志与追踪 | trace id、correlation id、日志等级、敏感信息处理 |
| 监控与告警 | 指标、健康检查、告警阈值、SLO |
| 配置 | 配置来源、动态开关、灰度、默认值 |
| 并发 | 线程模型、任务队列、锁、背压、资源隔离 |
| 数据一致性 | 事务、事件、幂等、版本、快照、最终一致性 |
| 接口规范 | API 版本、兼容性、超时、分页、限流 |
| 测试策略 | 单元测试、集成测试、契约测试、混沌测试 |
| 可演进性 | 模块边界、插件机制、迁移策略、废弃策略 |

**通用规约**：

- 第 8 章写规则和机制，不写零散实现。
- 只写对该系统重要、影响多个构建块、容易不一致的概念。
- 横切概念应被实现、测试和评审共同遵守。

### 3.10 第 9 章：架构决策

**目的**：记录重要、昂贵、风险高或影响长期演进的架构决策。

推荐轻量 ADR 格式：

```markdown
### ADR-001 决策标题

Status: Proposed / Accepted / Deprecated / Superseded

Context:
说明背景、约束、质量目标和冲突力量。

Decision:
说明最终选定的方案。

Consequences:
说明正面、负面和中性后果。
```

**进入 ADR 的标准**：

- 改变系统结构。
- 改变关键质量属性。
- 引入或移除关键依赖。
- 改变外部接口或数据模型。
- 改变构建、部署、安全或权限模型。
- 对团队协作或运维方式产生长期影响。
- 未来团队很可能追问“为什么这么做”。

### 3.11 第 10 章：质量要求

**目的**：把质量目标具体化、可验证化。

Q42 质量模型可作为质量维度参考：

| 维度 | 含义 |
| --- | --- |
| reliable | 在不中断或失败的情况下执行功能 |
| flexible | 适应需求、上下文或系统环境变化 |
| efficient | 在时间、容量、资源约束内完成工作 |
| usable | 让用户安全、有效、高效地使用 |
| secure | 保护数据并抵御攻击模式 |
| safe | 避免危及生命、健康或环境的状态 |
| maintainable | 可分析、修改、测试和演进 |
| suitable | 满足相关方明确和隐含需要 |
| operable | 易部署、易操作、易监控、易控制 |

**质量场景格式**：

| 字段 | 说明 |
| --- | --- |
| 场景 | 场景名称 |
| 背景 | 系统处于什么环境或状态 |
| 刺激 | 谁或什么触发了行为 |
| 响应 | 系统应该如何响应 |
| 指标 | 如何验证响应是否达标 |
| 优先级 | P0/P1/P2 或 Must/Should/Could |

**示例**：

```text
当外部依赖服务连续 30 秒不可用时，系统在 2 秒内进入降级路径，核心功能仍可用，并记录告警。
```

### 3.12 第 11 章：风险与技术债

**目的**：记录已知风险、问题、技术债和缓解措施。

**风险表**：

| 风险 | 触发条件 | 影响 | 缓解措施 | 验证方式 | Owner |
| --- | --- | --- | --- | --- | --- |

**通用规约**：

- 风险必须有缓解措施或明确接受理由。
- 高影响风险必须有 owner。
- 风险应与质量场景、架构决策和验证计划关联。
- 技术债应说明偿还条件，而不是只记录问题。

### 3.13 第 12 章：术语表

**目的**：统一业务和技术词汇，避免同义词、缩写和领域词造成歧义。

**通用规约**：

- 所有缩写必须解释。
- 领域词、状态名、角色名、事件名、接口名必须解释。
- 中英文混用时给出对应关系。
- 同一概念只保留一个推荐名称。

## 4. 通用使用规范

### 4.1 文档分级

| 等级 | 适用场景 | 章节要求 |
| --- | --- | --- |
| S 级：轻量方案 | 局部模块调整、低风险重构、小型功能设计 | 保留 1、2、3、4、5、6、9、11、12，可简写 7、8、10 |
| M 级：标准方案 | 跨模块、跨系统、核心流程、重要能力建设 | 1 到 12 全部保留，可裁剪深度 |
| L 级：高风险方案 | 安全、权限、数据一致性、稳定性、部署升级、重大架构演进 | 1 到 12 全部完整，并补质量场景、风险验证和 ADR |

### 4.2 必填规则

任何架构文档都应满足：

- 第 1 章说明目标、非目标、质量目标和相关方。
- 第 2 章说明约束及其影响。
- 第 3 章说明系统边界和外部接口。
- 第 4 章说明顶层策略，并映射质量目标。
- 第 5 章至少有 Level 1 构建块视图。
- 第 6 章至少有主成功路径和关键异常路径。
- 第 7 章说明构建块如何映射到运行环境。
- 第 8 章记录本系统最重要的横切规则。
- 第 9 章记录关键架构决策。
- 第 10 章包含可验证质量场景。
- 第 11 章包含风险、缓解措施、验证方式。
- 第 12 章解释缩写和核心术语。

### 4.3 裁剪规则

允许裁剪，但必须显式说明：

```markdown
本章不适用：原因是……
```

不允许直接删除章节，导致读者无法判断是遗漏还是不适用。

### 4.4 图表规则

arc42 不绑定图形记法。团队可以使用 C4、UML、Mermaid、白板图、表格或文本。选择图表形式时遵循“让读者更快理解”的原则。

| 信息类型 | 推荐形式 |
| --- | --- |
| 系统边界 | 上下文图 + 外部接口表 |
| 静态结构 | 构建块图 + 职责表 |
| 时序流程 | 时序图、编号步骤、事件流 |
| 状态流转 | 状态机、状态表 |
| 部署结构 | 部署图、运行单元表、目录树 |
| 决策 | ADR 表或 ADR 小节 |
| 风险 | 风险表 |

### 4.5 评审规则

架构评审按以下问题检查：

1. 是否说明为什么做。
2. 是否说明不做什么。
3. 是否列出真实约束。
4. 是否定义系统边界。
5. 是否识别外部参与者和接口。
6. 是否能看出静态结构和模块职责。
7. 是否能看出关键运行时流程。
8. 是否能看出部署和运行形态。
9. 是否集中描述安全、错误、日志、配置、并发、数据一致性等横切规则。
10. 是否解释关键决策原因和后果。
11. 是否有可验证质量场景。
12. 是否列出风险、缓解措施和验证方式。
13. 是否统一术语。

## 6. 通用最佳实践

### 6.1 先写第 1 到 4 章，再进入细节

先明确目标、约束、上下文和策略，再展开构建块、运行时和部署。这样能避免一开始陷入局部实现。

### 6.2 第 5 章只展开有架构意义的构建块

构建块视图的价值是抽象，不是罗列所有类、函数或文件。Level 1 必须有，Level 2/3 只展开关键、复杂、风险高或需要评审的部分。

### 6.3 第 6 章优先覆盖异常路径

架构问题经常隐藏在异常路径里：超时、依赖失败、并发冲突、状态不一致、重试风暴、版本不兼容、权限失败、数据损坏。运行时视图必须覆盖这些场景。

### 6.4 第 8 章写规则，不写散文

横切概念应沉淀为系统规则。例如：

- 每个外部请求必须具备可关联的追踪标识。
- 跨边界错误必须转换为稳定错误模型。
- 所有外部依赖调用必须有超时。
- 配置变更只影响新请求，或必须明确影响已运行任务。
- 敏感数据默认不进入普通日志。

### 6.5 第 9 章记录取舍和后果

架构决策不能只写最终结论，还要写背景和后果。尤其要记录负面后果和未来迁移成本。

### 6.6 第 10 章用场景替代形容词

不要写：

```text
系统要高性能、高可用、易维护。
```

要写：

```text
当每秒请求量达到目标峰值时，核心接口 p95 响应时间不超过指定阈值。
```

```text
当单个外部依赖不可用时，系统在指定时间内进入降级路径，核心功能仍可用。
```

### 6.7 第 11 章让风险可管理

风险必须有 owner、缓解措施和验证方式。无法消除的风险也要明确接受条件。

### 6.8 模板服务于表达，不反过来绑架表达

arc42 的价值是让架构信息有稳定位置。不要为了填满模板而制造内容，也不要为了简短而删除读者必须知道的信息。
